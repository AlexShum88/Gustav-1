# `core_v2`: план нового ядра

## Навіщо потрібен окремий модуль

Старе бойове ядро вже накопичило занадто багато зв'язків між симуляцією, представленням і debug-логікою.  
`core_v2` будується як окремий шар з такими принципами:

- авторитативний серверний цикл;
- клієнт лише відправляє команди інтерфейсу й отримує snapshot;
- доменні сутності не залежать від сцени або HUD;
- 3D-представлення ізольоване від правил бою;
- нові системи додаються поверх стабільного каркаса, а не всередину legacy-коду.

## Поточний стан цього етапу

У проєкт уже додано базу `core_v2`:

- `src/core_v2/application`
  серверний цикл, локальний клієнтський міст, фабрика test sandbox і генерація snapshot.
- `src/core_v2/domain`
  армія, бригада, батальйон, командир, наказ і стратегічний пункт.
- `src/core_v2/presentation`
  окремий 3D sandbox, камера top-down, battlefield view і новий HUD.
- `scenes/core_v2/CoreV2Sandbox.tscn`
  окрема сцена для запуску нового ядра без залежності від старої бойової сцени.

## Архітектурна схема

1. `CoreV2SandboxController`
   приймає input гравця, керує HUD і 3D-view.
2. `CoreV2ClientBridge`
   перетворює дії UI на команди для сервера.
3. `CoreV2Server`
   єдиний авторитативний вузол, що тікає fixed tick.
4. `CoreV2BattleState`
   зберігає поточний стан армій, штабів, батальйонів і стратегічних пунктів.
5. `CoreV2SnapshotBuilder`
   готує дані для клієнта без прямого доступу UI до доменних сутностей.
6. `CoreV2BattlefieldView`
   лише рендерить snapshot і не містить бойових правил.

## Етапи імплементації

1. План робіт і розрив з legacy.
   Уже зроблено: нове ядро винесено в окремий модуль.
2. Сервер-клієнтна база.
   Уже зроблено: є server-authoritative loop і client bridge.
3. Каркас сутностей.
   Уже зроблено: армія, бригада, батальйон, командири, штаби, обоз, strategic point.
4. Тестова мапа.
   Уже зроблено: 10x10 км sandbox з тиловими зонами й трьома селами.
5. Тестові бойові групи.
   Уже зроблено: дві армії, по дві бригади в тестовому сценарії.
6. LOS і туман війни.
   Частково зроблено: додано `CoreV2VisibilitySystem`, server-side фільтр enemy snapshot і last-seen markers. Висота рельєфу, дим і погодні модифікатори ще попереду.
7. Формації.
   Частково зроблено: додано `CoreV2FormationSystem`, sprite-offsets батальйону, поступову перебудову та 3D-рендер sprite blocks. Колони по дорогах і перерахунок після втрат ще попереду.
8. Рух.
   Уже є базовий server-side рух HQ і батальйонів по слотах, але без складних формацій і доріг.
9. Накази й доставка наказів.
   Частково зроблено: є палітра наказів і server-side призначення наказу бригаді. Доставка гінцями ще попереду.
10. Бій.
   Наступний крок: окремий combat layer v2 на рівні sprite executor.
11. Села, каравани, обози.
   Частково зроблено: обоз і стратегічні пункти вже присутні як доменні сутності, конвої ще не додані.

## Що свідомо НЕ переносилось зі старого ядра

- legacy `simulation` не використовується в `core_v2`;
- старі battle scene і HUD не підключались;
- старі системи формацій та combat не копіювались, щоб не перенести проблеми продуктивності;
- стару камеру не переносив, бо `core_v2` одразу піднято як 3D top-down sandbox.

## Найближчий технічний крок

Найбільш логічний наступний спринт для `core_v2`:

1. Ввести `terrain_height_field` і `terrain_region` v2.
2. Додати блокування LOS рельєфом, димом і погодою.
3. Додати messenger system для доставки наказів між штабами.
4. Додати movement planner для колон і руху дорогами.
5. Вести фактичний журнал рішень і обмежень у `docs/core_v2_journal_uk.md`.
